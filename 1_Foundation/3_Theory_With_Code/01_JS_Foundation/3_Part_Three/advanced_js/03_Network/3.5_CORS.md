# Fetch: Cross-Origin Requests (CORS) Summary

CORS (Cross-Origin Resource Sharing) is a security mechanism that controls cross-origin requests. By default, requests to another domain/protocol/port fail unless the server explicitly allows them.

## The Origin Concept

An **origin** = protocol + domain + port (e.g., `https://javascript.info:443`)

```javascript
// This fails because of different origin
try {
  await fetch("http://example.com");
} catch (err) {
  console.log(err); // Failed to fetch
}
```

## Historical Background

Before modern CORS, sites used workarounds:

### JSONP (JSON with Padding)

```javascript
// 1. Declare callback function
function gotWeather(data) {
  console.log(`Temperature: ${data.temperature}`);
}

// 2. Create script tag with callback parameter
let script = document.createElement("script");
script.src = "http://another.com/weather.json?callback=gotWeather";
document.body.append(script);

// 3. Server responds with: gotWeather({temperature: 25, humidity: 78})
```

## Safe Requests

A request is **safe** if it meets both conditions:

| Condition   | Allowed Values                                                                                                                          |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **Method**  | GET, POST, HEAD                                                                                                                         |
| **Headers** | Accept, Accept-Language, Content-Language, Content-Type (only `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain`) |

### Safe Request Flow

```javascript
// Safe request example
fetch("https://api.example.com/data", {
  method: "POST",
  headers: {
    "Content-Type": "application/x-www-form-urlencoded",
  },
  body: "name=John&age=30",
});
```

**Browser adds `Origin` header automatically:**

```
GET /data
Host: api.example.com
Origin: https://mysite.com
```

**Server must respond with:**

```
200 OK
Access-Control-Allow-Origin: https://mysite.com
```

or

```
Access-Control-Allow-Origin: *
```

### Accessing Response Headers

By default, only safe headers are accessible:

- Cache-Control, Content-Language, Content-Length, Content-Type, Expires, Last-Modified, Pragma

To expose additional headers:

```
Access-Control-Expose-Headers: Content-Encoding, API-Key
```

## Unsafe Requests (with Preflight)

Requests with:

- Non-safe methods (PUT, PATCH, DELETE)
- Non-safe headers (custom headers like `API-Key`)
- Non-safe Content-Type (`application/json`)

### Preflight Flow

```javascript
// This triggers preflight
fetch("https://api.example.com/service", {
  method: "PATCH",
  headers: {
    "Content-Type": "application/json",
    "API-Key": "secret123",
  },
  body: JSON.stringify({ status: "updated" }),
});
```

**Step 1 - Preflight (OPTIONS request):**

```
OPTIONS /service
Host: api.example.com
Origin: https://mysite.com
Access-Control-Request-Method: PATCH
Access-Control-Request-Headers: Content-Type, API-Key
```

**Step 2 - Preflight response:**

```
200 OK
Access-Control-Allow-Origin: https://mysite.com
Access-Control-Allow-Methods: PATCH, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, API-Key
Access-Control-Max-Age: 86400
```

**Step 3 - Actual request** (same as safe request)

**Step 4 - Actual response** (must include CORS headers)

## Credentials (Cookies, HTTP Auth)

By default, cross-origin requests don't send credentials.

### Sending Credentials

```javascript
fetch("https://api.example.com/user", {
  credentials: "include", // Send cookies and auth
});
```

### Server Response for Credentials

```
200 OK
Access-Control-Allow-Origin: https://mysite.com  // NOT *
Access-Control-Allow-Credentials: true
```

## CORS Headers Summary

| Header                             | Direction | Purpose                                  |
| ---------------------------------- | --------- | ---------------------------------------- |
| `Origin`                           | Request   | Browser sends the requesting origin      |
| `Access-Control-Allow-Origin`      | Response  | Server specifies allowed origin(s) or \* |
| `Access-Control-Allow-Methods`     | Response  | Allowed HTTP methods (preflight)         |
| `Access-Control-Allow-Headers`     | Response  | Allowed custom headers (preflight)       |
| `Access-Control-Expose-Headers`    | Response  | Headers readable by JavaScript           |
| `Access-Control-Allow-Credentials` | Response  | Allow cookies/auth                       |
| `Access-Control-Max-Age`           | Response  | Cache preflight response (seconds)       |
| `Access-Control-Request-Method`    | Request   | Preflight: requested method              |
| `Access-Control-Request-Headers`   | Request   | Preflight: requested custom headers      |

## Complete Example

```javascript
// Client-side code
async function patchData() {
  try {
    let response = await fetch("https://api.example.com/data/123", {
      method: "PATCH",
      headers: {
        "Content-Type": "application/json",
        "X-API-Key": "abc123",
      },
      credentials: "include", // Send cookies
      body: JSON.stringify({ status: "active" }),
    });

    let data = await response.json();
    console.log(data);

    // Access custom response header
    let version = response.headers.get("X-API-Version");
    console.log(version);
  } catch (err) {
    console.log("CORS error or network issue");
  }
}
```

**Required server response:**

```
OPTIONS /data/123
200 OK
Access-Control-Allow-Origin: https://mysite.com
Access-Control-Allow-Methods: PATCH, GET
Access-Control-Allow-Headers: Content-Type, X-API-Key
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 3600

PATCH /data/123
200 OK
Access-Control-Allow-Origin: https://mysite.com
Access-Control-Allow-Credentials: true
Access-Control-Expose-Headers: X-API-Version
X-API-Version: 2.0

{"id":123,"status":"active"}
```

## Task: Why Origin is Needed?

**Question:** Why is `Origin` header needed when `Referer` has even more information?

**Answer:**

1. **Privacy**: `Referer` contains the full URL (including path, query parameters, sometimes sensitive data). `Origin` only contains protocol/domain/port.

2. **Reliability**: `Referer` can be missing or modified (due to HTTPS→HTTP transitions, user settings, etc.). `Origin` is guaranteed to be present for cross-origin requests.

3. **Security**: `Origin` cannot be modified programmatically, making it reliable for CORS decisions.

4. **Not always present**: `Origin` is only sent for cross-origin requests. `Referer` may be absent when navigating from HTTPS to HTTP, or when user privacy settings disable it. Both can be missing in certain scenarios (bookmarks, direct navigation).
