# Error Handling: try...catch

## Overview

`try...catch` allows handling runtime errors without stopping the script. It catches errors that occur during code execution.

## Basic Syntax

```javascript
try {
  // Code to try
} catch (err) {
  // Error handling
}
```

### Example

```javascript
try {
  alert("Start of try");
  lalala; // Error: variable not defined
  alert("End of try (never reached)");
} catch (err) {
  alert(`Error: ${err.name}`); // Error: ReferenceError
}
```

## Important Limitations

### 1. Only Runtime Errors

`try...catch` cannot handle syntax errors (parse-time errors):

```javascript
try {
  {{{{{{{{{{{{
} catch (err) {
  alert("Won't work");  // Code is invalid, can't even run
}
```

### 2. Synchronous Only

`try...catch` does not work with scheduled code:

```javascript
try {
  setTimeout(() => {
    noSuchVariable; // Error not caught
  }, 1000);
} catch (err) {
  alert("Won't catch");
}

// Fix: Put try...catch inside the scheduled function
setTimeout(() => {
  try {
    noSuchVariable;
  } catch (err) {
    alert("Caught!");
  }
}, 1000);
```

## Error Object

Built-in errors have these properties:

| Property  | Description                                    |
| --------- | ---------------------------------------------- |
| `name`    | Error type (e.g., "ReferenceError")            |
| `message` | Human-readable error details                   |
| `stack`   | Call stack (non-standard but widely supported) |

```javascript
try {
  lalala;
} catch (err) {
  alert(err.name); // ReferenceError
  alert(err.message); // lalala is not defined
  alert(err.stack); // Stack trace
  alert(err); // ReferenceError: lalala is not defined
}
```

### Optional catch Binding

```javascript
try {
  // ...
} catch {
  // No (err) needed
  // Handle error without error object
}
```

## throw Operator

Create custom errors:

```javascript
throw <error object>

// Examples
throw new Error("Something went wrong");
throw new SyntaxError("Invalid syntax");
throw new ReferenceError("Variable not found");
```

### Custom Error Example

```javascript
let json = '{ "age": 30 }';

try {
  let user = JSON.parse(json);

  if (!user.name) {
    throw new SyntaxError("Incomplete data: no name");
  }

  alert(user.name);
} catch (err) {
  alert("JSON Error: " + err.message);
  // JSON Error: Incomplete data: no name
}
```

## Rethrowing Pattern

Catch only known errors, rethrow others:

```javascript
let json = '{ "age": 30 }';

try {
  let user = JSON.parse(json);

  if (!user.name) {
    throw new SyntaxError("Incomplete data");
  }

  blabla(); // Unexpected error
} catch (err) {
  if (err instanceof SyntaxError) {
    alert("JSON Error: " + err.message);
  } else {
    throw err; // Rethrow unknown errors
  }
}
```

## finally Clause

Runs always – after `try` (if no error) or after `catch` (if error):

```javascript
try {
  alert("try");
  if (confirm("Error?")) BAD_CODE();
} catch (err) {
  alert("catch");
} finally {
  alert("finally"); // Always runs
}
```

### finally with return

```javascript
function func() {
  try {
    return 1;
  } catch (err) {
    /* ... */
  } finally {
    alert("finally"); // Runs before return
  }
}

alert(func()); // finally, then 1
```

### try...finally (No catch)

Use when we don't want to handle errors but need cleanup:

```javascript
function func() {
  try {
    // Start something that needs cleanup
  } finally {
    // Cleanup runs even if error occurs
  }
}
```

## Global Error Handler (Browser)

```javascript
window.onerror = function (message, url, line, col, error) {
  alert(`${message}\n At ${line}:${col} of ${url}`);
};

function readData() {
  badFunc(); // Uncaught error
}

readData(); // Global handler catches it
```

## Summary

| Construct        | Purpose                        |
| ---------------- | ------------------------------ |
| `try...catch`    | Handle runtime errors          |
| `throw`          | Generate custom errors         |
| `finally`        | Code that always runs          |
| `window.onerror` | Global error handler (browser) |

**Key Points:**

- `try...catch` only catches runtime errors (not syntax errors)
- Errors in scheduled code need separate `try...catch`
- Use `throw` for custom errors
- Rethrow unknown errors to avoid hiding bugs
- `finally` runs regardless of error or return

## Task: Finally or Just the Code?

```javascript
// With finally
try {
  work work
} catch (err) {
  handle errors
} finally {
  cleanup
}

// Without finally
try {
  work work
} catch (err) {
  handle errors
}
cleanup
```

**Difference:** If there's a `return` inside `try` or `catch`, `finally` still runs, while code after `try...catch` does not:

```javascript
function withFinally() {
  try {
    return 1;
  } finally {
    alert("This runs before return"); // Executes
  }
}

function withoutFinally() {
  try {
    return 1;
  } catch (err) {}
  alert("This never runs"); // Not executed
}
```
