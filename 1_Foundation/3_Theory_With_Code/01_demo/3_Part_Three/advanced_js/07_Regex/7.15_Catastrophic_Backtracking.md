# Catastrophic Backtracking Summary

Some regular expressions can cause excessive backtracking, leading to CPU spikes and script hangs. This occurs when quantifiers create exponential numbers of possible matches.

## The Problem: Exponential Backtracking

```javascript
// This regexp can hang on certain inputs!
let regexp = /^(\d+)*$/;
let str = "012345678901234567890123456789z";

// May cause CPU 100% and script hang
console.log(regexp.test(str));
```

### Why Does It Happen?

For string `123456789z` (n=9 digits), the engine tries **511 combinations** to split digits:

- `(123456789)`
- `(12345678)(9)`
- `(1234567)(89)`
- `(1234567)(8)(9)`
- `(123456)(789)`
- `(123456)(78)(9)`
- …and so on

For n=20 digits: ~1 million combinations
For n=30 digits: ~1 billion combinations

The engine tries every possible way to group digits before discovering the final `z` makes it invalid.

## Common Pattern That Causes Issues

```javascript
// Pattern: words with optional spaces
let regexp = /^(\w+\s?)*$/;

// This string causes hanging due to many combinations
let str =
  "An input string that takes a long time or even makes this regex hang!";
```

Each `\w+` can split into multiple smaller words, creating exponential combinations.

## Solutions

### 1. Rewrite to Reduce Combinations

Make mandatory elements that prevent ambiguous splitting:

```javascript
// ❌ Problematic: optional space allows too many combinations
let regexp = /^(\w+\s?)*$/;

// ✅ Fixed: space is mandatory, reducing combinations
let regexp = /^(\w+\s)*\w*$/;
let str =
  "An input string that takes a long time or even makes this regex hang!";

console.log(regexp.test(str)); // false (works fast)
```

### 2. Prevent Backtracking with Lookahead (Emulate Possessive Quantifiers)

JavaScript doesn't support possessive quantifiers (`++`), but can emulate with lookahead:

```javascript
// Without backtracking prevention
let regexp = /^(\w+\s?)*$/; // slow

// With lookahead to prevent backtracking
let regexp = /^((?=(?<word>\w+))\k<word>\s?)*$/; // fast

let str =
  "An input string that takes a long time or even makes this regex hang!";
console.log(regexp.test(str)); // false (works fast)
```

**How it works:**

- `(?=(?<word>\w+))` - lookahead captures entire word
- `\k<word>` - matches that entire word (no backtracking)
- Prevents splitting word into smaller parts

## Visual Comparison

### Problematic Pattern

```
^(\w+\s?)*$
Each \w+ can be split: input → in|pu|t → multiple combinations
```

### Fixed Pattern

```
^(\w+\s)*\w*$
Space is mandatory, so words must be separated by spaces
```

## Practical Example

```javascript
// Problematic: tries all combinations
let badRegexp = /^(\d+)*$/;
let badString = "123456789z";

// Fixed: removes ambiguity
let goodRegexp = /^\d+$/; // simple, no backtracking issues
let goodString = "123456789";

console.log(goodRegexp.test(goodString)); // true (fast)
console.log(goodRegexp.test(badString)); // false (fast)
```

## Warning Signs

Your regexp might have catastrophic backtracking if:

1. Contains nested quantifiers: `(a+)*`, `(b*)*`, `(c+)+`
2. Has overlapping groups with optional elements: `(\w+\s?)*`
3. Performance degrades significantly with longer strings
4. CPU spikes to 100% on certain inputs

## Prevention Checklist

| Issue                               | Solution                                             |
| ----------------------------------- | ---------------------------------------------------- |
| Nested quantifiers `(a+)*`          | Rewrite to `a+` or use atomic grouping via lookahead |
| Optional elements causing ambiguity | Make elements mandatory when possible                |
| Too many splitting possibilities    | Redesign pattern to reduce combinatorial explosion   |
| Word splitting `(\w+\s?)*`          | Use `(\w+\s)*\w*` instead                            |

## Summary

**Catastrophic backtracking occurs when:**

- Quantifiers can split matches in many ways
- The regexp must try all combinations before failing

**Solutions:**

1. **Rewrite regexp** - make patterns more deterministic
2. **Use lookahead emulation** - prevent backtracking with `(?=(\w+))\1`
3. **Simplify** - remove unnecessary quantifiers

**Key Principles:**

- Avoid nested quantifiers like `(a+)*`
- Make optional elements mandatory when possible
- Use `^(\w+\s)*\w*$` instead of `^(\w+\s?)*$`
- Test with long strings containing non-matching characters
- For complex patterns, consider using a parser instead of regex
