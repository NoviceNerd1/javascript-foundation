# Comments in JavaScript

## Overview

Comments describe **how** and **why** code works. JavaScript supports:

- **Single-line**: `// comment`
- **Multiline**: `/* comment */`

## Bad Comments (What to Avoid)

### Don't Explain "What" the Code Does

```javascript
// This code will do this thing and that thing
very;
complex;
code;
```

**Problem:** Code should be self-descriptive. If it needs explanation, rewrite it instead.

### Recipe: Factor Out Functions

Instead of commenting complex logic, extract it into well-named functions.

**Bad (with comment):**

```javascript
function showPrimes(n) {
  nextPrime: for (let i = 2; i < n; i++) {
    // check if i is a prime number
    for (let j = 2; j < i; j++) {
      if (i % j == 0) continue nextPrime;
    }
    alert(i);
  }
}
```

**Good (self-descriptive):**

```javascript
function showPrimes(n) {
  for (let i = 2; i < n; i++) {
    if (!isPrime(i)) continue;
    alert(i);
  }
}

function isPrime(n) {
  for (let i = 2; i < n; i++) {
    if (n % i == 0) return false;
  }
  return true;
}
```

### Recipe: Create Functions for Code Blocks

**Bad:**

```javascript
// here we add whiskey
for (let i = 0; i < 10; i++) {
  let drop = getWhiskey();
  smell(drop);
  add(drop, glass);
}

// here we add juice
for (let t = 0; t < 3; t++) {
  let tomato = getTomato();
  examine(tomato);
  let juice = press(tomato);
  add(juice, glass);
}
```

**Good:**

```javascript
addWhiskey(glass);
addJuice(glass);

function addWhiskey(container) {
  for (let i = 0; i < 10; i++) {
    let drop = getWhiskey();
    smell(drop);
    add(drop, container);
  }
}

function addJuice(container) {
  for (let t = 0; t < 3; t++) {
    let tomato = getTomato();
    examine(tomato);
    let juice = press(tomato);
    add(juice, container);
  }
}
```

## Good Comments (What to Include)

### 1. Describe Architecture

High-level overview of components, interactions, and control flow.

### 2. Document Function Usage with JSDoc

```javascript
/**
 * Returns x raised to the n-th power.
 *
 * @param {number} x The number to raise.
 * @param {number} n The power, must be a natural number.
 * @return {number} x raised to the n-th power.
 */
function pow(x, n) {
  // implementation
}
```

Editors like WebStorm use JSDoc for autocomplete and code-checking. Tools like [JSDoc 3](https://jsdoc.app) generate HTML documentation.

### 3. Explain Why a Solution Was Chosen

When multiple solutions exist, explain why this specific approach was used—especially if it's not the most obvious one.

### 4. Document Subtle or Counter-Intuitive Code

If code has tricky or non-obvious behavior, explain it.

## Summary

| Type              | What to Do                                                                                     |
| ----------------- | ---------------------------------------------------------------------------------------------- |
| **Good comments** | Architecture overview, function usage (JSDoc), why solutions were chosen, subtle code behavior |
| **Avoid**         | Comments explaining "what" the code does (make code self-descriptive instead)                  |

**Key Principle:** Code should be so simple and clear that it rarely needs comments. Use functions as "executable comments" to make code self-descriptive.
