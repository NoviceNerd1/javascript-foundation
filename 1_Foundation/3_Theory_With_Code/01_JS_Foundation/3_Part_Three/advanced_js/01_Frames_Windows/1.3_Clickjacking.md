# Clickjacking Attack Summary

**Clickjacking** is an attack where an evil page tricks users into clicking on a victim site by overlaying a transparent iframe over seemingly harmless elements.

## How It Works

1. User visits malicious page with attractive link/button
2. Evil page positions transparent `<iframe>` containing victim site (Facebook, Twitter, etc.) over that button
3. User clicks the visible button, but actually clicks the hidden iframe content
4. Unauthorized action performed (Like, Follow, etc.)

### Attack Demo Structure

```html
<style>
  iframe {
    position: absolute;
    top: 0;
    left: -20px;
    opacity: 0; /* Completely invisible */
    z-index: 1;
    width: 400px;
    height: 100px;
  }
</style>

<!-- Victim site in iframe -->
<iframe src="https://facebook.com"></iframe>

<!-- Visible decoy button -->
<button>Click to get rich now!</button>
```

## Defence Methods

### 1. X-Frame-Options Header (Server-side, Recommended)

```http
X-Frame-Options: DENY                    # Never allow frames
X-Frame-Options: SAMEORIGIN              # Allow only same origin
X-Frame-Options: ALLOW-FROM https://site.com  # Allow specific domain
```

⚠️ Must be HTTP header, not HTML meta tag.

### 2. Framebusting JavaScript (Weak)

```javascript
if (top !== window) {
  top.location = window.location; // Break out of frame
}
```

**Evil page counter-measures:**

```javascript
// Block navigation attempts
window.onbeforeunload = function () {
  return false; // Prevents framebusting
};
```

### 3. Sandbox Attribute

```html
<!-- Blocks top navigation -->
<iframe sandbox="allow-scripts allow-forms" src="page.html"></iframe>
```

### 4. Covering Layer Protection

```html
<style>
  #protector {
    position: absolute;
    top: 0;
    left: 0;
    width: 100%;
    height: 100%;
    z-index: 99999999;
    background: white;
  }
</style>

<div id="protector">
  <a href="/" target="_blank">Go to site</a>
</div>

<script>
  // Remove protector if not in frame
  if (top === window) {
    document.getElementById("protector").remove();
  }
</script>
```

### 5. SameSite Cookie Attribute

```
Set-Cookie: session=secret; SameSite=Lax
```

Prevents cookies from being sent when site is loaded in iframe from other origins.

## Defence Comparison

| Method          | Type           | Effectiveness | Limitations                          |
| --------------- | -------------- | ------------- | ------------------------------------ |
| X-Frame-Options | Server header  | ✅ Strong     | Must configure server                |
| Framebusting    | Client JS      | ❌ Weak       | Easily bypassed                      |
| Sandbox         | HTML attribute | ⚠️ Moderate   | Requires iframe control              |
| Covering layer  | Client CSS/JS  | ✅ Strong     | Complex implementation               |
| SameSite cookie | Server header  | ✅ Strong     | Only protects authenticated sessions |

## Best Practices

1. **Primary defence**: Use `X-Frame-Options: SAMEORIGIN` on pages that shouldn't be framed
2. **Additional protection**: Implement covering layer for pages that legitimately need framing
3. **Authentication cookies**: Always use `SameSite` attribute
4. **Critical actions**: Add confirmation dialogs for sensitive operations

## Attack Limitations

- **Keyboard input** is difficult to hijack (users notice missing visual feedback)
- **Only affects mouse/tap actions**
- **Requires precise CSS positioning** to align click targets
