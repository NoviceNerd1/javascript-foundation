# Ninja Code (Irony Alert)

## Overview

"Ninja code" refers to deliberately unreadable coding practices. This is **ironic/satirical** advice showing what **NOT** to do. Good developers write clear, maintainable code.

---

## "Ninja" Techniques to Avoid

### 1. Extreme Brevity

**Bad:** Using complex one-liners

```javascript
// Taken from a well-known library
i = i ? (i < 0 ? Math.max(0, len + i) : i) : 0;
```

**Problem:** Impossible to understand at a glance.

### 2. One-Letter Variables

**Bad:**

```javascript
let a, b, c; // What do these mean?
```

**Exception:** `i` in short loops is acceptable.

### 3. Cryptic Abbreviations

**Bad:**

```javascript
let lst; // list
let ua; // userAgent
let brsr; // browser
```

**Problem:** Forces readers to guess meanings.

### 4. Overly Abstract Names

**Bad:**

```javascript
let data, value, item, elem, obj; // Tells nothing about purpose
```

### 5. Similar Names

**Bad:**

```javascript
let date; // calendar date
let data; // information
```

**Problem:** Easy to confuse and create typos.

### 6. Inconsistent Synonyms

**Bad:**

```javascript
function displayMessage() {}
function showName() {}
function renderUser() {}
function paintScreen() {} // All do similar things
```

### 7. Reusing Variables

**Bad:**

```javascript
function ninjaFunction(elem) {
  // 20 lines with elem
  elem = clone(elem);
  // 20 more lines - now elem is a clone!
}
```

**Problem:** Variable meaning changes mid-function.

### 8. Meaningless Underscores

**Bad:**

```javascript
let _name;
let __value;
let _temp; // No consistent meaning
```

### 9. Overly Fancy Names

**Bad:**

```javascript
let superElement;
let megaFrame;
let niceItem; // Says nothing about what they do
```

### 10. Variable Shadowing

**Bad:**

```javascript
let user = authenticateUser();

function render() {
  let user = anotherValue(); // Shadows outer variable
  // ... later, programmer thinks user is still from authenticateUser()
}
```

### 11. Hidden Side Effects

**Bad:**

```javascript
function checkPermission() {
  // Modifies something outside!
  // Returns complex object, not boolean
}
```

**Problem:** Functions with `is...`, `check...` should not change anything.

### 12. Bloated Functions

**Bad:**

```javascript
function validateEmail(email) {
  // Checks email AND shows error AND asks to re-enter
  // Name suggests it only validates
}
```

---

## Summary: Good vs "Ninja" Practices

| "Ninja" Practice      | Good Practice                               |
| --------------------- | ------------------------------------------- |
| One-letter variables  | Descriptive names: `user`, `count`, `total` |
| Cryptic abbreviations | Clear names: `userList`, `browserName`      |
| Overly abstract names | Meaningful names: `authenticatedUser`       |
| Variable reuse        | Separate variables for separate purposes    |
| Hidden side effects   | Functions do one thing, name reflects it    |
| Shadowing variables   | Distinct names for different scopes         |
| Inconsistent prefixes | Consistent naming conventions               |

> ⚠️ **Note:** This article is **satirical**. Do the opposite for clean, maintainable code.
