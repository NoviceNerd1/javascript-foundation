# Introduction: Callbacks

## Overview

Asynchronous actions (like loading scripts) don't finish immediately. Callbacks allow us to execute code after an asynchronous operation completes.

## Asynchronous Script Loading

```javascript
function loadScript(src, callback) {
  let script = document.createElement("script");
  script.src = src;

  script.onload = () => callback(null, script);
  script.onerror = () => callback(new Error(`Script load error for ${src}`));

  document.head.append(script);
}
```

### Usage

```javascript
loadScript("/my/script.js", function (error, script) {
  if (error) {
    alert("Error loading script");
  } else {
    alert("Script loaded successfully");
    // Use functions from the loaded script
    newFunction();
  }
});
```

## Error-First Callback Convention

The standard pattern for callbacks:

- **First argument**: reserved for error (`null` if no error)
- **Subsequent arguments**: result data

```javascript
function loadScript(src, callback) {
  // ...
  script.onload = () => callback(null, script);
  script.onerror = () => callback(new Error(`Error loading ${src}`));
}
```

## Sequential Callbacks (Callback Hell)

Loading multiple scripts in sequence creates nested callbacks:

```javascript
loadScript("1.js", function (error, script) {
  if (error) handleError(error);
  else {
    loadScript("2.js", function (error, script) {
      if (error) handleError(error);
      else {
        loadScript("3.js", function (error, script) {
          if (error) handleError(error);
          else {
            // All scripts loaded
          }
        });
      }
    });
  }
});
```

This is called **"callback hell"** or **"pyramid of doom"**:

```
loadScript('1.js', function() {
  loadScript('2.js', function() {
    loadScript('3.js', function() {
      // ...
    });
  });
});
```

## Avoiding Pyramid with Named Functions

```javascript
loadScript("1.js", step1);

function step1(error, script) {
  if (error) handleError(error);
  else loadScript("2.js", step2);
}

function step2(error, script) {
  if (error) handleError(error);
  else loadScript("3.js", step3);
}

function step3(error, script) {
  if (error) handleError(error);
  else {
    // All scripts loaded
  }
}
```

**Downsides:** Code is scattered, harder to read, creates single-use functions.

## Summary

| Concept                  | Description                                                             |
| ------------------------ | ----------------------------------------------------------------------- |
| **Callback**             | Function passed to asynchronous code, executed when operation completes |
| **Error-first callback** | Standard pattern: first argument = error, rest = results                |
| **Callback hell**        | Deeply nested callbacks from sequential async operations                |
| **Solution**             | Promises (next chapter) provide cleaner syntax                          |

**Key Points:**

- Callbacks allow handling asynchronous operations
- Error-first callback is the standard convention
- Nested callbacks lead to "pyramid of doom"
- Promises and async/await provide better alternatives
